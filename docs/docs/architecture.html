Architecture — Palace-X

A deep dive into Palace-X internals. This guide covers the hierarchical graph structure, compressed quantization pipeline, Apple Silicon optimizations, topological reranking, and concurrent index updates.

System Overview

9 Core Crates

Palace-X is modular. Each crate handles a distinct responsibility: graph topology, quantization, GPU dispatch, storage layout, concurrency, and orchestration.

GRAPH

palace-graph

HNSW construction, layer assignment, neighbor selection with α-RNG pruning. Builds the hierarchical navigable small world graph structure.

QUANTIZATION

palace-quant

RaBitQ implementation: Hadamard rotation, bit-plane projection, multi-bit distance estimation. Handles 1-bit through 8-bit quantization modes.

TOPOLOGY

palace-topo

Ego-graph construction, persistent homology H₀/H₁ computation via boundary matrices, β₁ Betti number extraction for loop-density reranking.

STORAGE

palace-storage

Memory layout: ArcSwap snapshots for wait-free reads, DashMap for lock-free writes. Hot/cold tier tracking, metadata serialization, persistence layer.

CORE

palace-core

Safe Rust types, trait definitions, distance metrics (L2, cosine). Zero unsafe code. Core abstractions shared across the system.

ENGINE

palace-engine

High-level search API: greedy descent, beam search, optional float reranking. Orchestrates the three-phase pipeline end-to-end.

METAL

palace-metal

Metal compute shaders for batch L2 and cosine distance. StorageModeShared zero-copy. Runtime detection of Apple Silicon, fallback to CPU on other platforms.

NEON

palace-neon

ARM64 NEON SIMD kernels: vtstq_u32 masked bit-plane sums, vcntq_u8 popcount, Hamming distance. Branchless implementations for consistent latency.

API

palace-x (root)

Public crate. Exports HnswRaBitQ builder, search interface, configuration. Unified namespace for end users.

Hierarchical Navigation

HNSW Graph Structure

Palace-X uses a Hierarchical Navigable Small World graph with Vamana-style α-RNG neighbor selection. The graph enables logarithmic-time descent to find nearby vectors, dramatically narrowing the search space before expensive distance computations.

PARAMETERS

Configuration

M = 16 Max neighbors per node (layer-0)
M_L = M Max neighbors (higher layers)
ef_c = 200 Beam width during construction
α = 1.2 RNG pruning parameter
LAYER ASSIGNMENT

Level Distribution

New vector insertion samples an entry level from:

level = floor(-ln(U) / ln(M))

where U is uniform [0,1). This ensures exponential decay of node count per layer, enabling O(log N) navigable path lengths.

α-RNG Pruning: During insertion, candidate neighbors are filtered using a Relative Neighborhood Graph criterion with parameter α. A candidate C is kept iff for all other candidates D: d(new, C) ≤ α × max(d(new, D), d(D, C)). This reduces hub formation while maintaining connectivity.

Concurrency & Snapshots

Writes use DashMap for lock-free concurrent neighbor list updates. Reads snapshot the entire graph via ArcSwap — a wait-free (no spinning) atomic pointer that serves immutable clones to searchers. This eliminates read-write contention during live search.

Wait-free Search: At most one insertion waits (publishing snapshot). All searches proceed concurrently without locks. Auto-publishes every 1000 inserts to amortize snapshot overhead.
Compressed Distances

RaBitQ: Randomized Binary Quantization

RaBitQ (SIGMOD 2024) projects full-precision vectors into 1–8 bit-planes via randomized Hadamard rotation. Similarity is estimated from weighted bit-plane inner products, achieving 4× memory compression with minimal recall loss.

ROTATION

Fast Hadamard Transform

Vectors undergo Walsh-Hadamard rotation H before binarization. This randomized projection preserves inner products in expectation and spreads information across bits.

x' = H × x

Complexity is O(D log D) via recursive butterfly operations, not O(D²).

BIT-PLANES

Multi-Bit Projection

After rotation, each vector component's sign bits are extracted into 4 planes (configurable 1–8). Plane k contains bit k across all dimensions.

plane_k[i] = (x'[i] >> k) & 1

Distance Estimation

The estimated inner product (cosine or L2) is computed from the bit-plane popcount sums:

Weighted Bit-Plane Inner Product:
IP_est = Σk=1B wk × popcount(plane_k(q) ⊙ plane_k(x))

where wk are learned weights (typically 2-k), ⊙ is bitwise AND, and B is the number of bit-planes (4 for 4-bit mode).

NEON Implementation

The critical kernel uses ARM64 NEON intrinsics for branchless masked sums: 

vtstq_u32(plane_k_q, plane_k_x)  // AND + test
vcntq_u8(...)                       // popcount
vmlaq_u32(sum, popcount, weight_k)  // MAC accumulate

Achieves 2.79M ops/s for 4-bit (384-dim) and 7.60M ops/s for 1-bit, with consistent sub-microsecond latency.

Why Multi-Bit? Single-bit quantization is fast but loses too much information (75.6% recall on SIFT-10K). 4-bit provides a sweet spot: 95%+ recall with 8× memory savings and still faster than float L2 in beam search.
Search Strategy

Three-Phase Pipeline

Phase 1: Greedy Descent

Starting from the entry point (top layer), we traverse the HNSW graph using float L2 distance. Each layer, we greedily move to the neighbor closest to the query. This O(log N) traversal narrows candidates from all N vectors down to ~100–200 promising neighbors.

Phase 2: RaBitQ Beam Search

The candidates from Phase 1 seed a beam search over the entire layer-0 graph. Distances are estimated using 4-bit RaBitQ, which is 3–5× faster than float L2 and reduces cache pressure. We expand the beam with the top matches, exploring their neighbors with compressed distances. This is where the bulk of evaluations happen, but at drastically reduced cost.

Phase 3: Topological Reranking (Optional)

If enabled, the top-K candidates from Phase 2 are reranked using a second signal: the β₁ Betti number (persistent homology). This penalizes hub nodes and promotes locally connected clusters. The rerank is O(K²) with K ≤ 100, negligible overhead.

Why This Gives HNSW Recall at RaBitQ Memory

The key insight is asymmetric pruning:

  • Phase 1 uses full precision to eliminate far-away regions.
  • Phase 2 uses cheap 4-bit estimates to rank remaining candidates.
  • Phase 3 (optional) uses topology to break ties among similar vectors.

Because Phase 2 operates only on vectors already deemed "close" by Phase 1, the 4-bit approximation error is small. We recover HNSW's accuracy without storing full-precision vectors for all N candidates. Result: 99.9% recall, 8× memory compression, competitive QPS.

Hardware Specialization

Apple Silicon & Unified Memory

M1/M2/M3 chips expose a unified memory architecture (UMA) with a single coherent address space for CPU and GPU. Palace-X exploits this with zero-copy Metal compute shaders and cache-aware data layout.

UMA MODEL

Shared Address Space

No discrete GPU memory. Buffers marked StorageModeShared are accessible from both CPU and GPU cores without explicit transfer. Eliminates PCIe bottleneck.

METAL SHADERS

Batch Compute

For large candidate sets (16K+), Metal achieves 1.4× speedup over NEON. Kernel computes L2 / cosine distance for all candidate-query pairs in parallel.

Cache Hierarchy & Prefetch

UMA memory is not equal: caches favor GPU-friendly access patterns. Palace-X lays out hot vectors (frequently visited in HNSW traversal) contiguously and cold vectors separately. CPU prefetch instructions (prfm) hint the cache subsystem.

Hot/Cold Tier Layout: After ~1000 insertions, vectors are sorted by access frequency. Top 10% occupy a compact "hot" region; remainder in "cold" region. HNSW navigation prefers hot vectors, reducing cache misses.

StorageModeShared Zero-Copy

Metal buffers are allocated with StorageModeShared: 

let buffer = device.newBuffer(bytes: data, length: size, options: .storageModeShared)
// CPU reads/writes to buffer.contents()
// GPU reads/writes via device.makeCommandBuffer()
// No memcpy() between CPU/GPU

The CPU and GPU cache lines coordinate automatically. This is unique to Apple Silicon and gives a decisive memory bandwidth advantage over discrete GPUs (PCIe ≈ 16 GB/s vs. UMA ≈ 100 GB/s).

Advanced Ranking

Persistent Homology & Betti Numbers

Topological reranking uses persistent homology to detect and penalize hub-like nodes. A vector that is "close" to many others may have low actual relevance. The β₁ Betti number quantifies local cluster density — high values indicate a rich topology, suggesting a "hub" node.

Ego-Graph Construction

For each candidate in the top-K from Phase 2:

  1. Build the neighborhood induced by edges among top-K and their HNSW neighbors.
  2. Compute a distance matrix d(·,·) for the ego-graph.
  3. Extract the (persistent) H₀ (components) and H₁ (loops) homology groups.

Betti Number Extraction

The 1st Betti number β₁ counts independent loops in the ego-graph. High β₁ means the candidate is surrounded by a rich, interconnected cluster. We use this as a secondary scoring signal:

Rerank Score:
score = (RaBitQ distance) + λ × β₁(candidate)

where λ is a tuning parameter. Candidates with high β₁ are penalized, promoting boundary nodes (good representatives of local structure).

Total Distance Metric

The complete metric combines quantized distance and topological penalty:

d_total(q, x) = d_rabitq(q, x) + λ × (β₁(x) − β₀(q))

β₀ is the number of connected components (usually 1 or close to 1). β₁ penalizes over-connected hub nodes. This is cheap to compute (O(K²) for K ≤ 100) and empirically improves precision@K.
Live Indexing

Concurrency & Auto-Publishing

Palace-X supports concurrent inserts and searches without blocking. New vectors can be added while queries are in flight. Consistency is eventual: inserts are visible after a snapshot publish.

DASHMAP

Lock-Free Writes

DashMap is a concurrent hash map with fine-grained locking. Neighbor lists are stored in DashMap entries, allowing multiple inserts to proceed in parallel without global locks.

ARCSWAP

Wait-Free Reads

ArcSwap is an atomic smart pointer enabling wait-free reads. Searches clone the current index snapshot with zero spinning or lock contention.

Insert Protocol

When a new vector is inserted:

  1. Sample entry level and assign initial neighbors via DashMap writes.
  2. Traversal links layer-by-layer; updates DashMap concurrently.
  3. Increment a publish counter.
  4. When counter reaches 1000, publish snapshot: ArcSwap(DashMap.clone()).
Auto-Publish Every 1000 Inserts: Snapshot publishing is expensive (requires cloning the entire DashMap). Auto-publishing amortizes this cost: 1000 inserts pay for 1 snapshot, ~0.1% overhead per insert. Searches always see recent state (≤1000 inserts old).

Search Protocol

Searches proceed without locks:

  1. Acquire snapshot via arc_swap.load() (wait-free, no spinning).
  2. Perform greedy descent and beam search on the snapshot.
  3. Release snapshot (reference count decrements).

If an insertion is publishing a snapshot concurrently, the insertion may block briefly. All searches proceed unaffected.

Memory Consistency

ArcSwap uses atomic release/acquire semantics to ensure that a search sees all inserts that were completed before the snapshot was taken. New inserts after the snapshot are buffered in DashMap; they appear in the next published snapshot.

Consistency Model: Writes are immediately visible to future reads. Reads see a snapshot of all writes completed before the snapshot epoch. This is "eventual consistency" with low latency: < 1ms for snapshot publish.
